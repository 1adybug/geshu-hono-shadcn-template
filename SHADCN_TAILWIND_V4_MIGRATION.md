# Codex 执行规约：将 geshu-next-template 衍生项目迁移到 shadcn/ui + Tailwind CSS 4

## 任务定位

你是负责在目标仓库中实际完成迁移的 Codex。本文件不是面向开发者的教程，也不是让用户手动复制命令和配置的操作手册。

收到本文件后，应先检查目标项目的真实状态，再直接实施迁移、修复迁移中出现的问题、执行验证并汇报结果。除非存在必须由用户决定的兼容性冲突，否则不要只给建议、输出示例代码或把执行步骤交还给用户。

本次迁移固定目标如下：

- 从 Ant Design + Tailwind CSS 3 迁移到 shadcn/ui + Tailwind CSS 4。
- 使用 shadcn 预设 `b6HGg41FS`。
- 按当前 shadcn CLI 的实际语义启用可点击按钮的 `pointer` 光标。
- 表单统一迁移到 TanStack Form，并继续复用目标项目已有的 Zod schema 与 parser。
- 后台采用常规的左侧菜单、右侧内容布局；桌面使用侧栏，移动端使用抽屉。
- 使用 TanStack Table 与 shadcn Table 构建业务层 DataTable。
- 支持系统、浅色和深色主题，默认跟随系统。
- 完整移除已经没有业务引用的 Ant Design、Tailwind CSS 3 及其配套依赖和样式。
- 保留目标项目现有的服务端行为、鉴权语义、公开首页、数据库结构、API、Server Action 和业务能力。

除上述目标和目标项目自身的业务约束外，所有可能随时间、CLI 版本、依赖版本、Next.js 版本或预设内容变化的信息，都必须在执行时实时获取。禁止从本文件、模型记忆或其他已经迁移的项目复制生成后的 CSS Token、组件源码、依赖版本和浏览器版本。

## 一、指令优先级与安全边界

### 1. 先读取目标项目规则

开始任何修改前，必须：

1. 查找并完整读取作用于目标文件的所有 `AGENTS.md`。
2. 读取用户当前指令，确认迁移范围、是否允许运行开发服务、是否要求提交。
3. 检查 Git 工作区和当前分支，识别用户已有改动。
4. 读取目标项目的包管理、构建、格式化、Lint 和供应链策略。

优先级为：用户当前明确指令、目标项目适用的 `AGENTS.md`、目标项目现有代码约定、本执行规约。

不得覆盖、回滚、格式化或提交与本次迁移无关的用户改动。不得使用 `git reset --hard`、`git checkout --` 或其他破坏性方式清理工作区。

### 2. 不扩大业务变更范围

迁移 UI 技术栈不代表可以修改服务端语义。除非用户明确要求，否则必须保持以下内容不变：

- 数据库结构和 Prisma 数据模型。
- Server Action、Shared Function 和 API Route 的对外行为。
- 请求参数、响应结构和 Zod schema 的业务规则。
- Better Auth 或目标项目实际鉴权方案的登录与会话语义。
- 管理员权限和普通用户权限。
- 公开 `/` 路由及其可访问性。
- 用户导入、导出、模板下载和文件响应行为。
- 查询缓存、缓存失效和服务端错误处理。

如果迁移过程中发现原有实现存在业务缺陷，应先区分它是迁移阻塞问题还是独立问题。非阻塞缺陷不要借迁移之名擅自改变。

### 3. 不启动开发服务

默认只执行静态检查和生产构建。未经用户或目标项目规则明确允许，不要运行 `pnpm dev`、`next dev` 或其他持续占用端口的服务。

如果运行时验证确实必要，应先获得授权；验证完成后关闭由本次任务启动的进程。

执行时必须通过目标项目、当前 CLI、本地 Next.js 文档和官方文档确认：

- 当前 shadcn CLI 支持的命令、参数、模板、预设解析和组件注册表。

- 当前预设实际生成的全局 CSS、字体、圆角、颜色、图标库和直接依赖。
- Tailwind CSS 4 当前针对 Next.js 的安装方式和浏览器最低要求。
- Next.js 当前对 PostCSS、全局 CSS、App Router、Server/Client Component 和构建环境的要求。
- shadcn 当前推荐的 Next.js 主题、TanStack Form、Sidebar、Data Table、日期选择和通知方案。
- Base UI 或其他底层组件当前的 API 和可访问性要求。

需要确认 Next.js 行为时，优先读取目标项目安装包中的 `node_modules/next/dist/docs/`。需要外部资料时，只使用官方一手来源，例如：

- <https://tanstack.com/table/latest/docs>

不要使用旧博客、搜索摘要或另一个项目的历史提交替代当前事实。

### 2. 禁止写死的内容

不得从本文件或其他迁移结果复制以下内容：

- `app/globals.css` 中的主题 Token。
- `components/ui` 中的组件源码。
- 预设当前对应的颜色、字体、圆角数值和暗色主题明细。
- 依赖的精确版本号。
- Tailwind CSS 4 的最低浏览器版本。
- shadcn CLI 当前参数和组件名清单。
- Next.js 当前的兼容矩阵和配置细节。

预设代码 `b6HGg41FS` 是固定目标，预设产生的具体文件内容不是固定快照。

### 3. 不伪造框架或项目结构

目标项目是 Next.js 项目时，应使用当前 CLI 对 Next.js 的正式支持。不要为了让 CLI 通过检测而伪造 Vite、Rsbuild、React Router 或其他框架文件。

如果当前 CLI 不能安全初始化现有项目，或会覆盖用户已有业务文件，可以在系统临时目录创建一个当前 CLI 支持的最小 Next.js 参考项目，用它获取本次执行时的准确预设结果。参考项目只用于辨别 CLI 生成内容和依赖，不能成为目标项目的结构来源。

清理临时目录前，必须解析并核对绝对路径，确认其位于预期临时根目录内。

## 三、迁移前项目审计

### 1. 检查仓库状态

至少执行：

```powershell
git status --short
git branch --show-current
git remote -v
```

如果项目存在 `template` 远程，可以用它辅助理解项目来源，但不得直接合并模板或覆盖项目定制内容，除非用户同时明确要求同步模板。

### 2. 读取基础配置

根据目标仓库实际文件读取：

- `package.json`、锁文件和 `pnpm-workspace.yaml`。
- `next.config.*`、`tsconfig.json` 和环境变量示例。
- `postcss.config.*`、`tailwind.config.*` 和 `app/globals.css`。
- 根布局、路由组布局、全局 Registry 或 Provider。
- ESLint、Prettier、Browserslist 和 Git 忽略配置。
- Prisma schema、迁移目录和生成配置。
- 鉴权配置、请求封装、客户端 instrumentation 和错误页。

不能假设所有 `geshu-next-template` 衍生项目仍与当前模板目录完全一致。必须确认项目是否使用 `src`、是否调整了路由组、是否新增组件库、是否保留自定义 Tailwind 插件，以及锁文件是否被跟踪。

### 3. 建立旧技术栈清单

使用 `rg` 搜索实际引用，不要只看 `package.json`。搜索范围至少包括：

- `antd`、`@ant-design/*`、`soda-antd`。
- `ConfigProvider`、`StyleProvider`、全局 `message` 和 `.ant-*` 样式。
- `soda-tailwind`、Tailwind CSS 3 指令和旧插件。
- `@tabler/icons-react` 或项目实际使用的旧图标库。
- 旧字体包及字体入口。
- Ant Design Form、Modal、Drawer、Table、DatePicker、Select 和 Upload。
- 被封装在公共组件中的 Ant Design 间接引用。

对每个依赖和引用分类：

- 仅服务于旧 UI，可在迁移完成后删除。
- 被业务逻辑独立使用，必须保留。
- 迁移期间需要暂时共存。
- 只存在于文档或历史文本，不影响运行时。

例如，日期库可能同时被表单、格式化和服务端代码使用，不能因为 Ant Design 被删除就自动删除。

### 4. 记录关键业务链路

迁移前应定位并记录：

- 登录、初始化首个用户、退出和鉴权重定向。
- 个人资料、手机号验证码和用户编辑。
- 用户新增、封禁、解封、删除、导入和导出。
- 操作日志、错误日志和 JSON 详情。
- 系统设置、自动备份和限流配置。
- URL 查询参数驱动的筛选、排序和分页。
- `schemas -> shared -> actions -> presets -> hooks` 或目标项目实际采用的调用链。

这些链路是迁移验收基线，不能只以页面能渲染为完成标准。

### 5. 运行迁移前基线检查

在不改变仓库状态的前提下，执行目标项目已有的类型检查、Lint、格式检查和构建。若项目使用 Prisma，先执行当前项目要求的客户端生成命令。

如果生产构建依赖密钥或 URL：

- 优先使用当前 PowerShell 进程内的临时占位环境变量进行验证。
- 不要把临时密钥写入 `.env`、`.env.local` 或提交文件。
- 如果不能安全提供占位值，应记录为可复现的环境阻塞，而不是假装构建通过。

保存迁移前已有错误，避免把历史问题误判为本次迁移引入。

### 1. 检查当前 CLI

先查看当前 shadcn CLI 总帮助，以及初始化、添加组件和预设相关子命令的帮助。根据实际输出确定：

- `pointer` 是布尔开关、带值参数还是已更名。
- 预设解码、解析和应用命令。
- 是否支持预览、差异检查和覆盖控制。

不要机械照抄历史命令。即使命令看起来熟悉，也要先验证当前帮助信息。

### 2. 解析 `b6HGg41FS`

使用当前 CLI 提供的预设解码或等价命令获取本次执行时的真实配置，并保存输出以供最终校验。

至少核对：

- 组件风格。
- 基础色和主题色。
- 图表色。
- 图标库。
- 正文和标题字体。
- 圆角档位。
- 菜单颜色和强调方式。
- `pointer` 行为。

如果当前解析结果与用户指定代码不一致，停止使用历史认知，先判断 CLI 版本、预设服务或命令是否变化。

### 3. 初始化策略

优先让当前 shadcn CLI 在目标 Next.js 项目中初始化预设，但必须先检查它将修改的文件。

如果初始化会覆盖已有 `app/globals.css`、组件目录或别名配置：

1. 先记录用户原有内容。
2. 使用 CLI 当前提供的预览能力，或在临时 Next.js 参考项目中生成。
3. 区分预设生成内容、框架模板内容和目标项目业务样式。
4. 只把当前预设与必要基础设施安全合并回目标项目。

初始化完成后立即检查 Git 差异，确认没有意外生成第二套 `src`、`app`、`lib` 或错误别名。

## 五、Tailwind CSS 4 与 Next.js 基础设施

### 1. 采用当前官方 Next.js 接入方式

打开执行时的 Tailwind CSS Next.js 官方安装文档和目标项目本地 Next.js 文档，按当前方案配置 Tailwind CSS 4。

必须确认：

- Tailwind CSS 4 的编译入口。
- 当前是否通过 PostCSS 插件接入。
- `postcss.config.*` 的当前有效格式。
- 全局 CSS 应在何处导入。
- 当前是否还需要独立的 Tailwind 配置文件。

不要因为当前模板的迁移结果使用某一种 PostCSS 配置，就假设未来项目仍然相同。

### 2. 迁移旧 Tailwind 配置

读取旧 `tailwind.config.*`，逐项处理：

- 自定义主题和设计 Token。
- 插件。
- Content 扫描路径。
- Safelist。
- 动态类名。
- 第三方包源码扫描。
- Ant Design 与 Tailwind 的层级兼容配置。

只有确认旧文件不再承担职责后才删除。不能因为模板默认配置简单，就直接删除衍生项目中的定制配置。

动态拼接的 Tailwind 类名应改为可静态分析的完整类名。真正运行时才能确定的尺寸、坐标、颜色和资源 URL，可以使用 CSS 变量或 `style`。

### 3. 合并全局 CSS

预设主题必须来自本次 CLI 的实际生成结果。合并时：

- 保留目标项目仍需要的业务级全局样式。
- 清理 Tailwind CSS 3 指令和已失效的层级声明。
- 在对应 Ant Design 组件迁移后，删除 `.ant-*` 覆盖。
- 保留根节点高度、滚动行为和打印样式等非 UI 库规则。
- 确认亮色和暗色 Token 都完整。
- 确认 `pointer` 行为符合当前 CLI 或当前预设实现。

自定义 CSS 不能只通过源码存在来判断有效。生产构建后还应确认它没有因层级、扫描或优化规则被裁剪。

### 4. 浏览器兼容性

执行时读取 Tailwind CSS 官方兼容性页面，并与目标产品要求比较。

如果产品允许，提高 `.browserslistrc` 或等价配置到不低于当前 Tailwind CSS 4 的最低要求。如果产品必须支持更旧浏览器，不得擅自降低产品兼容范围；应向用户报告冲突，并说明继续使用 Tailwind CSS 3 或调整浏览器支持范围的取舍。

禁止把其他项目曾使用的浏览器版本直接复制过来。

## 六、shadcn 本地配置与生成组件

### 1. 配置真实路径

根据目标项目实际结构创建或更新 `components.json`。对于仍采用本模板根目录结构的项目，优先保持以下约定：

- 全局 CSS 位于 `app/globals.css`。
- CLI 原始组件位于 `components/ui`。
- shadcn 工具位于 `utils/shadcn.ts`。
- 公共业务组件位于 `components`。
- Hooks 位于 `hooks`。
- 工具别名为 `@/utils/shadcn`。

这些是目录约定，不是要求无条件复制的配置快照。如果目标项目使用 `src` 或其他别名，应按其真实结构调整，并确保 TypeScript、Next.js 和 shadcn CLI 三者解析一致。

Tailwind 配置字段、RSC 开关和 schema 字段必须符合执行时的当前 shadcn 规范。

### 2. 只通过 CLI 添加原始组件

不得手写或直接修改 `components/ui/**/*.tsx`。根据目标项目实际用到的功能，使用当前 shadcn CLI 添加所需组件。

本模板家族通常需要覆盖以下能力，但必须以目标项目实际页面为准：

- 按钮、输入、文本域、选择器、开关和表单字段。
- Card、Badge、Avatar、Skeleton 和 Alert。
- Dialog、Alert Dialog、Popover、Dropdown Menu、Tooltip 和 Sheet。
- Sidebar、Table、Pagination、Calendar 和 Sonner。

先检查 CLI 的预览或差异能力，再添加组件。不要为了“以后可能用到”一次性安装整个注册表。

业务封装如 `DataTable`、`DateRangePicker`、`RoleSelect`、确认按钮和用户选择按钮应放在 `components`，不要放入 `components/ui`。

### 3. 生成代码忽略规则

如果目标项目规则允许，合并 ESLint 和 Prettier 忽略规则，使以下生成内容不被项目自定义格式反复改写：

- `components/ui/**`。
- `utils/shadcn.ts`。

必须先确认 ESLint 配置导出形式，再合并忽略项；不得覆盖已有忽略规则。使用 Prettier 文件信息和 ESLint 显式检查确认规则真实生效。

## 七、依赖迁移策略

### 1. 先迁移引用，再删除依赖

不要一开始就删除 Ant Design。推荐顺序：

1. 接入 Tailwind CSS 4 和 shadcn 基础配置。
2. 迁移全局 Provider、主题和消息系统。
3. 迁移公共组件和布局。
4. 迁移表单、表格、日期选择和弹窗。
5. 逐页清理旧引用。
6. 确认引用清零后删除旧依赖。

大型衍生项目可以短期共存两套 UI，但必须明确记录剩余范围，不能把阶段性共存报告为完整迁移。

### 2. 清理旧依赖

以目标项目实际引用为准，检查并清理：

- Ant Design 及其 Next.js 样式注册包。
- `soda-antd`。
- `soda-tailwind`。
- 仅服务于 Tailwind CSS 3 的依赖和配置加载器。
- 旧图标库。
- 旧字体包。
- 不再使用的 Autoprefixer 或其他 PostCSS 插件。

日期库、文件处理库、Query、Hooks 和服务端包如果仍被业务使用，必须保留。不要仅根据“旧模板曾经一起安装”进行批量删除。

### 3. 遵守 pnpm 供应链策略

读取目标项目的 `pnpm-workspace.yaml`、最小发布时间、构建脚本白名单和 lockfile 策略。新增依赖后按实际直接依赖更新允许列表，不要把另一个项目的白名单原样复制。

如果安装被供应链策略拒绝，应定位具体包、发布时间或构建脚本，不要关闭整个安全策略规避问题。

## 八、全局 Provider、主题与消息系统

### 1. 保留必要 Provider

重构 Registry 或根 Provider 时，应保留目标项目原有的数据能力，例如 TanStack Query 的 QueryClient 和默认请求策略。

移除只服务于 Ant Design 的：

- ConfigProvider。
- 样式 Registry 或 CSS-in-JS Provider。
- Ant Design locale。
- `globalThis.message` 注入。

根据当前 shadcn 官方 Next.js 方案接入主题 Provider、Tooltip Provider 和 Sonner Toaster。Provider 的嵌套顺序应满足实际上下文依赖，不要照搬历史代码。

### 2. 主题要求

主题至少支持：

- 跟随系统。
- 浅色。
- 深色。
- 用户选择持久化。

默认跟随系统。主题切换入口放在后台侧栏底部，以及登录、初始化和公开首页可访问的位置。避免在服务端和客户端首次渲染时直接读取未挂载的主题值，防止 hydration 不一致。

JSON 查看器、图表或其他拥有独立配色系统的组件也必须随当前解析主题切换。

### 3. 消息迁移

所有 Ant Design `message` 调用迁移为当前 Sonner API，覆盖：

- 加载中。
- 成功。
- 失败。
- 使用稳定标识更新同一条 Toast。
- 手动关闭。

请求错误、客户端 instrumentation 和 mutation preset 应直接使用 Sonner。不要创建 Ant Design 风格的全局兼容层，也不要继续写入 `globalThis.message`。

迁移 mutation preset 时必须保留原有 Query 缓存刷新和异常传播语义。

## 九、TanStack Form 迁移规范

### 1. 适用范围

登录、初始化、用户新增与编辑、封禁、手机号修改、验证码、查询筛选、个人资料和系统设置等表单统一使用 TanStack Form。

不要引入 React Hook Form、Zod Resolver 或模拟 Ant Design Form 的兼容封装。

### 2. 复用现有 Zod 模型

每个表单应：

- 使用完整业务 schema 作为提交校验来源。
- 对字段使用已有字段级 schema，在失焦或目标项目约定的时机校验。
- 通过 shadcn Field、FieldError、`data-invalid` 和 `aria-invalid` 展示错误。
- 保留原始必填、长度、格式、枚举和关联字段规则。

必须核对执行时 TanStack Form 对 Standard Schema 的行为。如果校验结果不会把 Zod 转换后的值交给 `onSubmit`，则在 `onSubmit` 中再次调用目标项目已有的 composite parser，以确保 `trim()`、格式标准化和其他转换真实生效。

不能只因为校验通过就直接把原始 `value` 传给 mutation。

### 3. 提交与状态

统一由表单提交事件阻止浏览器默认行为后调用 `form.handleSubmit()`，并通过 `form.Subscribe` 或当前推荐 API 订阅：

- `canSubmit`。
- `isSubmitting`。
- `isPristine`。

提交按钮应正确处理重复提交、mutation pending 和表单未修改状态。不要让对话框底部按钮脱离实际表单状态。

### 4. 异步默认值和弹窗复用

用户详情和系统设置等异步数据到达后，使用当前 TanStack Form 推荐的 reset 能力同步完整默认值，不能只在初始化时读取一次。

弹窗关闭时应根据业务语义重置：

- 表单值与错误。
- 验证码倒计时。
- 临时服务端错误。
- 快捷日期选择状态。
- mutation 残留状态。

### 5. 非提交操作

发送验证码、预检查和其他不触发表单提交的操作，应先用对应字段 parser 校验当前值，再调用 mutation。不能绕过 schema，也不能因为整个表单尚未填写完就阻止独立字段操作。

## 十、后台布局和页面迁移

### 1. 常规后台布局

使用当前 shadcn Sidebar 能力构建：

- 桌面端固定展开侧栏。
- 移动端抽屉或 Sheet。
- 右侧可滚动内容区。
- 统一页面标题、说明、操作栏和筛选区。
- 侧栏底部的主题与账户操作。

品牌链接返回公开首页。菜单包含个人中心，以及仅管理员可见的用户管理、操作日志、错误日志和系统设置。

不要为了视觉重构改变鉴权边界或公开路由语义。

### 2. 页面迁移范围

至少覆盖目标项目实际存在的：

- 公开首页。
- 登录页。
- 初始化页。
- 错误页和 Not Found。
- 个人中心。
- 用户管理。
- 操作日志和错误日志。
- 系统设置。

所有页面使用预设语义色，不要硬编码只适合亮色主题的颜色。加载、空数据、成功、失败、禁用和移动端状态都要迁移。

### 3. 个人中心与设置布局

个人中心使用 shadcn Card、Avatar、Badge、Field、Dialog 等重新组织信息和编辑操作。顶部人物信息区应形成完整、稳定的视觉块，避免装饰色只覆盖卡片局部高度造成割裂。

系统设置的各板块应根据自身内容决定高度。在宽屏下使用瀑布流或等价的独立高度分栏，避免 Grid 同行拉伸导致短卡片底部出现大片空白。窄屏回到单列，并避免 CSS 多列造成表单内容被拆分。

## 十一、表格、筛选和日期时间

### 1. 业务层 DataTable

新建或迁移业务层 `DataTable`，基于当前 shadcn Table 与 TanStack Table，实现：

- 手动服务端排序。
- 手动服务端分页。
- 加载骨架。
- 空状态。
- 总数和页码。
- 页容量切换。
- 宽表横向滚动。
- 稳定的操作列。

列定义、数据、加载状态、页码、页容量、总数、排序状态和对应回调应作为明确接口传入。不要把 API 请求耦合进通用 DataTable。

表头和普通单元格按当前产品要求居中。真正没有内容的普通单元格显示 `-`；业务状态如“未封禁”不是空值，必须由列渲染函数明确输出。

如果使用 `:empty::before` 统一补占位符，应验证：

- 选择器只命中真实空单元格。
- 不影响带隐藏节点、骨架或交互控件的单元格。
- 规则在 Tailwind CSS 4 生产构建后仍存在。
- 占位符使用语义弱化颜色。

不能用占位符掩盖列渲染逻辑漏掉了业务数据的问题。

### 2. 用户与日志能力

用户管理应保留：

- 新增、编辑、封禁、解封和删除确认。
- 模板下载、批量导入和导出。
- URL 驱动的筛选。
- 服务端排序与分页。
- 角色代码与中文展示名称的映射。

日志页面应保留筛选、排序、分页、文本详情和 JSON 参数弹窗。

### 3. 日期和时间本地化

日期选择组件应使用中文本地化，星期从周一到周日排列，并显示中文星期文本。格式化、占位符、范围说明和无障碍标签也应使用中文。

创建受控的业务层 DateRangePicker，并让所有日志和用户筛选复用，避免每个页面自行组合日期选择器。

封禁时间、日期范围和其他日期时间输入应审计一致性。若当前 shadcn 注册表没有完整日期时间选择器，可以基于当前 Calendar、Popover 和输入组件组合业务组件，或保留满足产品要求的原生输入；但必须统一本地化、校验、最小值、键盘操作和明暗主题体验，不能只因为原生控件省事而留下明显不一致。

封禁快捷项应保留常用时长和永久选项。永久状态下，“永久”按钮必须处于可感知的选中状态，并提供正确的 `aria-pressed` 或当前组件等价语义。

### 4. 筛选布局

筛选表单应根据字段宽度使用响应式 Grid 或 Flex，日期范围不能因为内部宽度或错误的跨列类名无条件独占整行。

应分别验证桌面和移动端：

- 桌面端字段紧凑排列，日期范围只占所需列数。
- 移动端字段自然换行并保持可点击区域。
- 查询和重置按钮位置稳定。
- URL 参数与表单状态同步。

## 十二、Base UI 与可访问性

shadcn 当前预设可能基于 Base UI，也可能在未来改变底层实现。必须遵循本次 CLI 生成组件的实际 API。

重点检查：

- Button 通过 `render` 渲染 Next.js Link 时的原生按钮语义。
- Select 是否需要显式 items 映射才能显示标签而不是内部代码。
- Dialog、Sheet 和 Dropdown 的焦点管理。
- Trigger 的 `aria-expanded`、`aria-invalid` 和键盘行为。
- 表单按钮的 `type`，避免在嵌套表单中误提交。

如果控制台报告“期望原生 button，但实际渲染为非 button”，应按当前 Button API 明确声明非原生按钮语义或改用真实按钮。不要关闭警告、修改生成组件或用展示层补丁掩盖。

## 十三、Next.js 服务端边界

### 1. 保持 Server/Client Component 边界

迁移交互组件时，只在实际需要状态、事件或浏览器 API 的文件中添加 `"use client"`。不要把整个页面树无条件转为 Client Component。

客户端组件不能导入 `server` 目录、密钥、Prisma 或仅服务端可用模块。服务端组件也不要为了使用一个按钮而整体客户端化，应把交互部分拆到适当的业务组件。

### 2. 保持既有调用链

如果项目采用以下链路，应继续保持职责：

- `schemas`：参数校验、类型和 parser。
- `shared`：服务端业务逻辑。
- `actions`：Server Action 薄包装。
- `presets`：mutation 默认提示、缓存刷新和副作用。
- `hooks`：客户端请求函数和 Hook。

UI 迁移不得把业务逻辑搬进页面，也不得绕过 shared 层直接访问 Prisma。

### 3. 路由和元数据

保留 App Router 路由组和鉴权布局。移动客户端交互代码时，检查页面 metadata 的放置位置，不要因为页面改成客户端组件而丢失标题和描述。

独立 API Route 只在文件下载、上传、流或现有 HTTP 能力确实需要时保留；不要因 UI 迁移新建重复接口。

## 十六、常见故障处理

### 1. CLI 部分写入后失败

先检查 Git 差异和包清单，确认 CLI 已经创建或修改了哪些文件。区分框架检测、网络、依赖策略、别名和配置 schema 错误。不要重复运行覆盖命令，导致无法辨别哪次写入有效。

### 2. 生成组件缺少依赖

以生成组件的真实导入和 TypeScript 模块缺失错误为依据补装直接依赖。不要从其他项目的 `package.json` 批量复制。

### 3. 预设解析不一致

重新解码预设，并对比本次参考项目与目标项目的：

- Style。
- Base Color 和 Theme。
- 字体与圆角。
- 图标库。
- 菜单配置。
- CSS 路径与 Token。

修复根因，不要只改一个表面变量来欺骗解析结果。

### 4. TanStack Form 转换未生效

检查提交时传给 mutation 的是原始字段值还是 parser 结果。验证包含空格的用户名、昵称、手机号或验证码，确认转换发生在业务请求之前。

### 5. Select 显示内部代码

检查当前 Select API、items 映射和 SelectValue 的取值方式。角色等枚举需要显示中文名称，同时提交稳定代码。不要把显示文本写回业务枚举，也不要修改生成组件。

### 6. Link 按钮出现原生语义警告

检查生成 Button 的 `nativeButton` 或当前等价属性，以及 `render` 的最终元素。Link 必须声明正确的非按钮语义，普通提交和操作仍应使用真实 button。

### 7. 空单元格占位符没有出现

先检查单元格是否真的为空，再检查 CSS 选择器是否在生产输出中存在。若列渲染函数错误返回空值，应修正业务渲染，而不是扩大 CSS 选择器。

### 8. 构建因鉴权密钥失败

确认这是生产构建的环境前置条件，而不是 UI 迁移错误。使用当前进程的非敏感占位值验证可构建性，禁止写入仓库或输出真实密钥。
